# API 执行概览仪表盘与失败诊断入口计划

## 目标

在现有“数据仪表盘”中补齐 API 执行概览，使用后端真实执行历史聚合数据，不使用假数据。第一版聚焦最近执行健康度、失败分布、待处理任务和失败诊断入口；点击失败记录后在仪表盘内打开右侧抽屉，展示失败详情，并支持跳转到 API 自动化页继续处理。

## 接口约定

- 新增只读接口：`GET /api/api-execution/dashboard/summary?project_id=&limit=50`
- 数据来源复用现有 SQLite `runs`、`automation_tasks`，不新增表，不做迁移。
- `limit` 默认 50，最大 200。
- 状态集合固定为 `queued`、`running`、`passed`、`failed`、`cancelled`。
- 失败原因优先取 `failure_reason`，没有则取失败步骤的 `error`，再没有归为“未知失败”。
- 失败步骤按 `method + url/name` 聚合。
- 待处理任务只统计 `status=pending`，并受 `project_id` 过滤。
- 返回 `template_stats`，按执行记录中的 `flow_template_id` 聚合模板执行数、通过率、失败率、失败数、平均耗时和最近执行时间。

## 前端交互

- `DashboardPage` 的 `API 执行概览` 分区替换“开发中”占位。
- 顶部展示执行总数、通过率、失败数、待处理数、平均耗时。
- 中部展示状态分布、失败原因 Top 5、失败步骤 Top 5。
- 新增“流程模板执行表现”表格，展示最近执行记录中带模板来源的模板通过率和失败率。
- 底部展示最近执行记录，支持项目筛选和刷新。
- 点击失败记录打开右侧诊断抽屉。

## 诊断入口

- 抽屉展示执行基本信息、失败原因、失败诊断、失败步骤、断言失败详情、请求/响应摘要。
- 抽屉提供“跳转 API 自动化”入口。
- 跳转时通过 `sessionStorage` 暂存 `run_id`，API 自动化页挂载后读取执行详情，载入脚本和执行报告，并进入结果诊断步骤。
- 第一版不在仪表盘内直接触发自动修复，修复仍在 API 自动化页完成。

## 测试清单

- 后端聚合测试覆盖混合状态、项目过滤、空历史、失败原因和失败步骤统计。
- 后端聚合测试覆盖模板维度通过率、失败率和失败数。
- 前端构建需通过 `npm run build`。
- 回归确认覆盖率仪表盘仍可嵌入显示，API 自动化页现有执行和结果诊断不受影响。

## 后续增强

- 增加时间范围筛选和分页总数。
- 将失败诊断抽屉中的“跳转 API 自动化”升级为更明确的“载入编辑并修复”操作。
- 增加项目/环境维度趋势图。
- 将待处理任务列表直接接入仪表盘。
